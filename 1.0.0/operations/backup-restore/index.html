    Backup & Restore

    Mium splits its persistent state across three storage tiers, and each tier needs its own backup strategy. There is no "one backup for everything" — restoring Mium is restoring three things in order.

    What Lives Where

    Tier Backend What It Holds Backup Cadence
    Bootstrap RocksDB embedded on each Master IAM (users / groups / policies / companies / orgs), KMS keys, ConnectionStore Daily snapshot of data/ on the leader Master
    Application data NeorunBase (PostgreSQL) Chat sessions, chat messages, saved prompts, embeddings Use NeorunBase's standard PG backup (pg_dump / WAL archiving)
    Export ciphertext S3-compatible bucket Server-rendered XLSX / PDF / PPTX (encrypted) S3 lifecycle / cross-region replication if needed; usually not backed up — files are short-lived

    The master key (MIUM_MASTER_KEY) is the secret that unlocks the KMS bundle. Without it, nothing in any tier is recoverable. Back the master key up in your secret manager — separately from the data — and treat it like a database master password.

    The Bootstrap RocksDB Tier

    The leader's data/ directory contains three RocksDB instances:

    data/
├── kms/            # Encrypted KEK bundle (versioned)
├── iam/            # Users, groups, policies, companies, orgs
└── connections/    # Per-user encrypted tool & LLM credentials

    Followers hold replicas of all three, but the leader is the source of truth. Back up from the leader to avoid catching a partial snapshot mid-replication.

    Hot Backup

    RocksDB exposes a checkpoint API that creates a consistent snapshot of the live data without stopping writes. The simplest operator-side approximation is a flush + filesystem copy:

    1. On the leader, send SIGTERM to drain in-flight writes, or quiesce traffic at the load balancer for ~5 seconds.
    2. rsync -a data/ /path/to/backup/data-$(date +%Y%m%d)/
    3. Resume traffic.

    This costs a brief read-only window. Operators who cannot accept that should run a pre-built backup tool against RocksDB's checkpoint API.

    Restore

    Restoring the bootstrap tier is destructive — it overwrites the cluster's identities and credentials. Coordinate the steps:

    1. Stop every Master and Worker. The cluster cannot be partially restored.
    2. Restore the master key into MIUM_MASTER_KEY on every node — this must match the value the backup was taken under.
    3. On every Master, replace data/ with the backup snapshot (one snapshot copied to all Masters; followers will overwrite from the leader on next start, but seeding all of them avoids a long initial sync).
    4. Workers do not carry RocksDB state — leave their data/ empty.
    5. Start the leader Master first, wait for /ready, then start the followers, then the Workers.

    NeorunBase Tier

    Chat sessions, messages, prompts, and embeddings live in NeorunBase tables under the mium.neorunbase.schema schema (default mium):

    • mium_chat_session
    • mium_chat_message
    • mium_prompt
    • mium_embedding

    Use NeorunBase's standard PostgreSQL-compatible backup — pg_dump for periodic dumps, WAL archiving for point-in-time recovery, or the platform's snapshot mechanism if it has one. There is nothing Mium-specific here.

    Schema Auto-Creation

    Mium creates these tables on startup if they don't exist. After a NeorunBase restore (or migration to a fresh cluster), simply restart any Master and the schema is materialised. Restoring from pg_dump includes the schema directly.

    Export Ciphertext Tier (S3)

    Server-rendered exports are stored as opaque ciphertext in the configured S3 bucket. They are short-lived by design:

    • The download endpoint deletes the object as soon as the download response completes.
    • The retention sweep (mium.tempfile.ttl.days, default 10 days) deletes any orphans.

    Backups are usually unnecessary — a lost export is regenerated by re-running the chat turn. If your compliance regime requires it, S3 cross-region replication or versioning is the right tool.

    Disaster Recovery Drill

    A recovery is only as good as the rehearsal. Quarterly drill:

    1. Provision a fresh empty cluster.
    2. Restore master key, then bootstrap RocksDB, then NeorunBase schema.
    3. Log in with a non-bootstrap user (the bootstrap admin should already be rotated — see Security) and verify chat history loads, saved prompts are present, and existing connections work.
    4. Run a SQL query end-to-end through Ontul.
    5. Render and download an XLSX export.

    If any step fails, the gap is in your backup runbook, not in Mium.

    What You Cannot Restore

    • Lost master key — every encrypted store becomes unreadable. Treat the master key with at least the same care as the bootstrap database password.
    • Partial-cluster snapshots — restoring the leader's RocksDB while followers run a different generation produces split-brain on the next KMS_SYNC. Always stop the whole cluster before restoring.
    • Time-travel for individual users — Mium does not version chat history at the row level. NeorunBase point-in-time recovery is the only path back.
